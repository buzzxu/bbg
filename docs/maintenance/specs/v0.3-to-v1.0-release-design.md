# BBG v0.3.0 → v1.0.0 Release Design

**Created**: 2026-03-31
**Status**: Approved
**Target**: npm publish as `@buzzxu/bbg-cli`
**Audience**: Open-source community

---

## Overview

This spec defines the complete path from v0.2.0 (foundation hardening, complete) to v1.0.0 (npm publish). The work is organized into three milestones, each with a clear theme and acceptance criteria.

### Current State (v0.2.0)

- 6 CLI commands: init, add-repo, doctor, sync, release, upgrade
- 25 agents, 60 skills, 34 rules, 32 commands (governance content)
- 201 tests passing, zero TypeScript errors, zero lint errors
- ESLint + Prettier + GitHub Actions CI configured
- Technical debt eliminated (10 duplicated functions extracted, init.ts split, bugs fixed)

### Key Decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Target audience | Open-source community | Full documentation, stable API, npm publish |
| Scaffold tier | Remove | Vestigial (1 file, 4 lines); rename to "two-tier template system" |
| Language coverage | Full for 5 mainstream (TS, Python, Go, Java, Rust); partial for Kotlin, PHP, C++ | Balance effort vs. value |
| Cross-references | Full 150+ docs | Agent↔skill↔rule↔command interlinking for navigation |
| Context files | Handlebars templates | Replace hardcoded paths with `{{variable}}` placeholders |
| npm package name | `@buzzxu/bbg-cli` | Scoped package, clear naming |
| Milestones | Three: v0.3.0, v0.4.0, v1.0.0 | Incremental releases, earlier usability |

#### 2. Context File Templatization

Convert `contexts/dev.md`, `contexts/review.md`, `contexts/research.md` from static files to Handlebars templates.

Current problem: files contain hardcoded bbg-specific paths (`src/utils/`, `src/constants.ts`), commands (`npm test`, `npm run build`), and tool references (`TodoWrite`).

Solution:
- Move to `templates/handlebars/contexts/dev.md.hbs`, `review.md.hbs`, `research.md.hbs`
- Replace hardcoded values with template variables from `TemplateContext`:
  - `{{buildCommand}}` — from stack info (e.g., `npm run build`, `go build`, `cargo build`)
  - `{{testCommand}}` — from stack info (e.g., `npm test`, `pytest`, `go test`)
  - `{{srcDir}}` — from structure analysis (e.g., `src/`, `pkg/`, `lib/`)
  - `{{projectName}}` — from config
  - `{{languages}}` — joined language list
- Update `TemplateContext` interface if needed to include `buildCommand`, `testCommand`, `srcDir`
- Update template manifest to use handlebars mode instead of copy mode for contexts
- Update `buildGovernanceManifest()` — contexts should no longer be plain copy tasks
- Remove the static `contexts/` files from governance copy list
- Existing static `contexts/` files become the fallback/default when no stack is detected

#### 3. Remove Scaffold Tier

## v0.4.0 — Capability Expansion

**Theme**: From "generates files" to "maintains them".
**Branch**: `feature/v0.4.0-capability-expansion`

### P0 — Must Have

#### 10. `bbg upgrade` Enhancement

Current: Basic diff-based upgrade with snapshot tracking. User-modified files get a unified patch shown but are not auto-merged.

Enhancement:
- **Three-way merge**: Compare (old template, current user file, new template) to auto-merge non-conflicting changes
- **Conflict markers**: When auto-merge fails, insert `<<<<<<< / ======= / >>>>>>>` markers like git
- **`--interactive` flag**: Step through each conflict for manual resolution
- **Preserve user customizations**: Track which files the user has intentionally modified vs. which are untouched from template

Implementation approach:
- Add a simple three-way merge algorithm (or use a lightweight library)
- Extend the snapshot hash system to track "user-modified" vs "template-pristine" status
- Update `upgrade.ts` to use three-way merge for user-modified files

#### 11. `bbg doctor --self` Self-Check Mode

Add a `--self` flag that validates the governance content itself (not the target project):

Checks:
- All agents referenced in commands exist
- All skills referenced in agents exist
- All rules referenced in skills exist
- Cross-reference links are valid (no broken links)
- Template manifest matches files on disk
- Governance file counts match documentation claims
- No orphaned files (files on disk not referenced by anything)

This validates the integrity of bbg's own governance content before deployment.

#### 12. Template Consistency Validation in Doctor

Extend `bbg doctor` (normal mode) to check:
- All files from the template plan exist in the target project
- No files are missing that should have been deployed
- Hash-tracked files haven't been corrupted
- Governance file versions match the bbg version that generated them

#### 13. `bbg release` Auto-Changelog

Enhance `bbg release` to:
- Parse git log since last release tag
- Group commits by conventional commit type (feat, fix, refactor, etc.)
- Generate a CHANGELOG entry in standard Keep a Changelog format
- Optionally append to `CHANGELOG.md`

### P1 — Should Have

#### 14. Plugin Architecture Foundation

Allow users to extend bbg with custom governance content:

- Plugin directory: `~/.bbg/plugins/` (global) or `.bbg/plugins/` (project-local)
- Plugin structure: same as bbg's governance dirs (`agents/`, `skills/`, `rules/`, `commands/`)
- Plugin discovery: scan plugin directories during `bbg init` and merge with built-in content
- Plugin manifest: `plugin.json` with name, version, description, content listing
- `bbg init` merges plugin content with built-in governance content

Scope for v0.4.0: basic plugin loading and merging. No plugin registry, no versioning, no conflict resolution.

### P2 — Nice to Have

#### 15. Multi-Agent Collaboration Patterns

Add skills for agent orchestration workflows:
- `skills/agent-handoff/SKILL.md` — How to hand off work between agents
- `skills/agent-pipeline/SKILL.md` — Sequential agent processing pipelines
- Update existing agent docs with collaboration patterns

## Risk Assessment

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|------------|
| Cross-reference mapping errors | Medium | Medium | Automated link validation in doctor --self |
| Three-way merge complexity | High | High | Start with simple algorithm, fall back to manual |
| Plugin architecture scope creep | Medium | High | Strict v0.4.0 scope: load + merge only |
| npm name availability | Low | High | `@buzzxu/bbg-cli` is scoped, very likely available |
| Breaking changes during v0.3-v0.4 | Medium | Medium | Semantic versioning, CHANGELOG discipline |

1. **v0.3.0 plan** — Execute first, mostly governance content work + config

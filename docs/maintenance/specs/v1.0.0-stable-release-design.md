# BBG v1.0.0 — Stable Release Design

**Created**: 2026-04-01
**Status**: Approved
**Target**: npm publish as `@buzzxu/bbg-cli`
**Branch**: `release/v1.0.0`

---

## Overview

v1.0.0 takes BBG from a working internal tool (v0.4.0) to a production-ready, published npm package. The work is organized into two phases: publishing infrastructure (make `npm pack` → install → `bbg init` work end-to-end) and documentation/quality (make the package presentable and maintainable for open-source users).

### Current State (v0.4.0)

- 6 CLI commands: init, add-repo, doctor, sync, release, upgrade
- 25 agents, 63 skills, 34 rules, 40 commands (governance content)
- ~251 tests, 88.10% overall coverage, zero TypeScript errors
- Three-way merge upgrades, doctor --self, auto-changelog, plugin architecture
- Template manifest 100% in sync with files on disk

### Key Decisions

| Decision            | Choice                                  | Rationale                                           |
| ------------------- | --------------------------------------- | --------------------------------------------------- |
| npm package name    | `@buzzxu/bbg-cli`                       | Scoped package, CLI binary remains `bbg`            |
| License             | AGPL-3.0-only                           | Already in LICENSE file                             |
| Author              | `buzzxu`                                | Copyright holder                                    |
| Publish method      | Local `npm pack` + manual `npm publish` | Simple, no CI/CD automation needed for v1.0         |
| Kotlin/PHP/C++ gaps | Deferred to post-1.0                    | Already marked "Partial" in Language Support Matrix |
| Programmatic API    | CLI-only for v1.0                       | `exports` field points to CLI entry, no library API |

### Task 3: API Stability Audit

Review and document public interfaces for stability:

1. **BbgConfig schema** (`src/config/schema.ts`):
   - Document all fields, their types, and whether they are required/optional
   - Verify no planned breaking changes
   - Mark `plugins` field as experimental

2. **CLI command signatures**:
   - Document all 6 commands with their flags/options
   - Verify flags are consistent and well-named
   - Mark any experimental flags

3. **Template variables** (`TemplateContext` interface):
   - Document all variables available in Handlebars templates
   - Ensure naming is consistent

4. **Governance directory structure**:
   - Document the convention: `agents/<name>.md`, `skills/<name>/SKILL.md`, `rules/<lang>/<name>.md`, `commands/<name>.md`

Audit results will be incorporated into README.md (Task 5) and do not require a separate document.

### Task 4: npm pack End-to-End Verification

Verification steps (executed as part of implementation, not a test file):

1. `npm run build` — ensure dist/ is fresh
2. `npm pack` — generate tarball
3. Extract tarball to temp directory
4. Verify file inventory:
   - `dist/cli.js` exists
   - `agents/*.md` — 25 files
   - `skills/*/SKILL.md` — 63 files
   - `rules/**/*.md` — 34+ files
   - `commands/*.md` — 40 files
   - `hooks/` — 8 files
   - `templates/` — template files present
   - `mcp-configs/` — 2 files
   - `contexts/` — 3 files
5. Install from tarball in a clean directory: `npm install <tarball>`
6. Verify `npx bbg --help` outputs command list
7. Verify `npx bbg --version` outputs correct version

**Note**: Full `bbg init` test requires interactive prompts, so `--help` and `--version` are sufficient for automated verification. Manual `bbg init` testing is recommended before actual `npm publish`.

## Out of Scope

- Kotlin/PHP/C++ command coverage completion (deferred to post-1.0)
- CI/CD automated publishing (manual npm publish for v1.0)
- Plugin registry or plugin versioning
- Programmatic library API (CLI-only for v1.0)
- GitHub Release creation (manual after npm publish)

# /ts-build

## Description

Resolve TypeScript/JavaScript build errors with language-specific strategies.

## Usage

```
/ts-build [error-description]
```

## Process

1. Run the build command (`npm run build` / `npx tsc --noEmit`) to reproduce the error
2. Read the full error output — identify the root cause (type error, import resolution, module system mismatch)
3. Classify the error:
   - **Type error** — Wrong type assignment, missing generic, incompatible interface
   - **Import error** — Missing `.js` extension (ESM), circular dependency, unresolved module
   - **Module error** — CJS/ESM interop, `moduleResolution` mismatch, missing `type: "module"`
   - **Build tool error** — tsup/esbuild/webpack config issue, entry point misconfiguration
4. Search the codebase for related patterns — how was this solved elsewhere?
5. Apply the minimal fix. Do not refactor surrounding code
6. Run the build again — confirm the original error is gone and no new errors appeared
7. Run the test suite — confirm no regressions

## Output

```
Error: [original error message]
Root cause: [classification + explanation]
Fix: [what was changed and why]
Verification: build passes, N tests pass
```

## Rules

- Fix the build error, nothing else. Do not refactor or improve adjacent code.
- For ESM projects, always use `.js` import extensions even for `.ts` source files.
- If a type error reveals a genuine bug (not just a type annotation issue), flag it but fix only the type error.
- If the fix requires a new dependency, justify it before installing.
- Always verify with both `npm run build` and `npm test` after the fix.

## Examples

```
/ts-build "Property 'branches' does not exist on type 'string[]'"
/ts-build "Cannot find module './utils/fs' — missing .js extension"
/ts-build "Module '"execa"' has no default export"
```
```

# /ts-test

## Description

Write and run TypeScript/JavaScript tests following TDD with vitest patterns.

## Usage

```
/ts-test [function-or-module]
```

## Process

1. Identify the function or module to test. Read its source to understand inputs, outputs, and edge cases
2. Create or locate the test file following the convention: `tests/unit/<path>/<module>.test.ts`
3. Write a failing test first (RED):
   - Import the function using `.js` extension for ESM compatibility
   - Use `describe`/`it` blocks with descriptive names
   - Cover the happy path first, then edge cases, then error cases
4. Run the test: `npx vitest run tests/unit/<path>/<module>.test.ts`
5. Verify it fails for the right reason (not a syntax error or import issue)
6. Implement the minimal code to make the test pass (GREEN)
7. Run the test again — confirm it passes
8. Refactor if needed (IMPROVE) — extract helpers, improve naming, reduce duplication
9. Run the full test suite to confirm no regressions

## Output

```
Tests written: N new test cases in <file>
Coverage: [functions/branches tested]
Status: all N tests passing
```

## Rules

- Always use `vitest` (not jest or mocha). Import from `vitest`: `describe`, `it`, `expect`, `vi`, `beforeEach`, `afterEach`.
- Use `.js` extensions in all imports within test files.
- Mock external dependencies (file system, network, child processes) — never make real I/O calls in unit tests.
- Use `vi.mock()` for module mocking, `vi.spyOn()` for partial mocking.
- Clean up temp files/dirs in `afterEach` hooks.
- Test file naming: `<module>.test.ts` — matches the source module name.
- Prefer `toBe` for primitives, `toEqual` for objects/arrays, `toContain` for substring/array membership.

## Examples

```
/ts-test src/utils/git-url.ts
/ts-test "the buildTemplateContext function"
/ts-test "error handling in renderProjectTemplates"
```

## Usage

```
/rust-build [error-description]
```

## Process

1. Run `cargo build` (or `cargo check` for faster feedback) to reproduce the error
2. Read the full error output — Rust errors include suggestions, read them carefully
3. Classify the error:
   - **Ownership/borrow** — Move after borrow, lifetime mismatch, dangling reference
   - **Type error** — Trait not implemented, type mismatch, missing conversion
   - **Module error** — Visibility (`pub`), module tree, unresolved import
   - **Dependency error** — Feature flag missing, version incompatibility, build script failure
4. Follow the compiler's suggestion first — `rustc` suggestions are usually correct
5. If the suggestion doesn't apply, search for the pattern in the codebase
6. Apply the minimal fix
7. Run `cargo build` and `cargo test` to verify

## Output

```
Error: [original error message]
Root cause: [classification + explanation]
Fix: [what was changed and why]
Verification: cargo build passes, N tests pass
```

## Rules

- Read the compiler error message completely, including the help/suggestion section.
- Fix the build error, nothing else.
- Prefer owned types (`String`, `Vec`) for return values; borrow (`&str`, `&[T]`) for parameters.
- If a lifetime annotation is needed, keep it as simple as possible — avoid named lifetimes unless the compiler requires them.
- Always verify with both `cargo build` and `cargo test`.

## Examples

```

## Rules

- Unit tests go in `#[cfg(test)] mod tests` at the bottom of the source file.
- Integration tests go in `tests/` directory.
- Use `#[test]` attribute, not a test framework crate (unless the project already uses one).
- For async tests, use `#[tokio::test]` if the project uses tokio.
- Use `mockall` or similar only if the project already depends on it.
- Test naming: `test_<function>_<scenario>` (e.g., `test_parse_url_empty_input`).

import { describe, it, expect } from "vitest";
import { existsSync } from "node:fs";
import { readFileSync, readdirSync } from "node:fs";
import { join, resolve } from "node:path";

const ROOT = resolve(__dirname, "../../..");

function getGovernanceFiles(): string[] {
  const files: string[] = [];

  // Agents
  for (const f of readdirSync(join(ROOT, "agents"))) {
    if (f.endsWith(".md")) files.push(join(ROOT, "agents", f));
  }

  // Skills
  for (const d of readdirSync(join(ROOT, "skills"))) {
    const skillFile = join(ROOT, "skills", d, "SKILL.md");
    if (existsSync(skillFile)) files.push(skillFile);
  }

  // Rules
  for (const subdir of readdirSync(join(ROOT, "rules"))) {
    const fullPath = join(ROOT, "rules", subdir);
    try {
      for (const f of readdirSync(fullPath)) {
        if (f.endsWith(".md") && f !== "README.md") {
          files.push(join(fullPath, f));
        }
      }
    } catch {
      // Not a directory
    }
  }

  // Commands
  for (const f of readdirSync(join(ROOT, "commands"))) {
    if (f.endsWith(".md")) files.push(join(ROOT, "commands", f));
  }

  return files;
}

describe("governance cross-references", () => {
  const files = getGovernanceFiles();

  it("all governance docs have a Related section", () => {
    const missing: string[] = [];
    for (const file of files) {
      const content = readFileSync(file, "utf8");
      if (!content.includes("## Related")) {
        missing.push(file.replace(ROOT + "/", ""));
      }
    }
    expect(missing).toEqual([]);
  });

  it("all relative links in Related sections point to existing files", () => {
    const brokenLinks: string[] = [];
    for (const file of files) {
      const content = readFileSync(file, "utf8");
      const relatedSection = content.split("## Related")[1];
      if (!relatedSection) continue;

      const linkPattern = /\]\(([^)]+)\)/g;
      let match;
      while ((match = linkPattern.exec(relatedSection)) !== null) {
        const linkPath = match[1];
        if (linkPath.startsWith("http")) continue;
        const resolvedPath = resolve(join(file, ".."), linkPath);
        if (!existsSync(resolvedPath)) {
          brokenLinks.push(`${file.replace(ROOT + "/", "")}: ${linkPath}`);
        }
      }
    }
    expect(brokenLinks).toEqual([]);
  });
});

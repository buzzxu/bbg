# Analyze 2.0 Phase A/B Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Deliver the first two Analyze 2.0 phases safely: first lock down the current analyze contract with tests and artifact responsibilities, then add the first-generation evidence and lifecycle substrate in an additive, backward-compatible way.

**Architecture:**

- **Phase A** is contract hardening. It adds no new user-visible behavior. It creates the regression net needed for later refactors.
- **Phase B** is substrate introduction. It adds stable item IDs, provenance, evidence indexing, and lifecycle metadata while preserving current `.bbg/knowledge/` consumption paths.

**Tech Stack:** TypeScript, Vitest, existing analyze orchestrator and CLI, current `.bbg/knowledge/` JSON files, current task routing, wiki artifact generation.

## Phase A/B Sequencing Notes

Recommended execution order:

1. A1
2. A2
3. A3
4. A4
5. B1
6. B2
7. B3
8. B4
9. B5
10. B6
11. B7
12. B8

Why this order:

- Phase A locks current behavior before structure changes begin
- B1/B2 establish the core language of the new model
- B3 creates evidence before emission is upgraded
- B4 upgrades current files first
- B5/B6 add normalized outputs after item identity exists
- B7 proves compatibility before moving on to later phases

---

## Scope Guardrails

Do **not** include in Phase A/B:

- full Socratic interview engine logic
- CLI UX redesign beyond what tests already cover
- Hermes candidate creation and promotion
- verification feedback loops
- autonomous LLM self-interview
- removal of legacy fields or current domain files

Phase A/B is complete when the current analyze contract is protected, the first-generation knowledge substrate exists, and all existing runtime consumers still work without requiring users to learn anything new.

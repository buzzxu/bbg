# Analyze 2.0 Implementation Roadmap

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this roadmap incrementally. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Upgrade `bbg analyze` from a deterministic artifact generator into an AI-native project understanding system that produces reusable knowledge, feeds Hermes, supports business-focused deep dives, and preserves a one-command user experience.

**Architecture:** Analyze 2.0 is a layered upgrade, not a rewrite. It preserves the current analyze pipeline, current CLI entry point, current `.bbg/knowledge/` and wiki outputs, and current Hermes foundations. New capabilities are introduced in stages: evidence extraction, Socratic questioning, AI-context execution preference, Hermes lifecycle integration, and finally promotion-quality feedback loops.

**Tech Stack:** TypeScript, Vitest, existing CLI/runtime pipeline, current `.bbg/knowledge/` JSON layout, existing wiki generation, Hermes SQLite schema and command system, optional LLM provider integration.

**Design Inputs:**

- `docs/superpowers/specs/2026-04-20-socratic-analyze-interview-design.md`
- `docs/superpowers/specs/2026-04-20-bbg-analyze-ux-design.md`
- `docs/superpowers/specs/2026-04-20-hermes-analyze-knowledge-lifecycle.md`

## Delivery Strategy

Analyze 2.0 should ship in **seven phases**:

1. **Phase A:** Stabilize the current analyze contract and artifact model
2. **Phase B:** Introduce evidence items and lifecycle metadata
3. **Phase C:** Add Socratic interview engine behind a feature flag
4. **Phase D:** Upgrade CLI UX to AI-first execution with low-friction handoff/auto-launch
5. **Phase E:** Connect analyze outputs into Hermes candidate lifecycle
6. **Phase F:** Add runtime back-reference and verification feedback loops
7. **Phase G:** Make the new path default and deprecate the legacy interview path

Each phase must be independently valuable, testable, and reversible.

## Phase A: Stabilize Current Analyze Contract

**Theme:** Freeze the surface before deepening the system.

**Goal:** Ensure current analyze outputs, current CLI behavior, and current runtime consumers are clearly defined and covered by tests before structural changes begin.

### Files likely involved

- `src/cli.ts`
- `src/commands/analyze.ts`
- `src/analyze/orchestrator.ts`
- `src/analyze/types.ts`
- `src/analyze/emit-knowledge.ts`
- `src/runtime/wiki.ts`
- `src/runtime/tasks.ts`
- test files covering analyze output and task knowledge consumption

### Tasks

- [ ] Add snapshot-style contract tests for analyze result shape and key knowledge files
- [ ] Add regression tests for focus query behavior using positional input
- [ ] Add regression tests for handoff-required behavior outside AI context
- [ ] Add regression tests for current task knowledge consumption from `.bbg/knowledge/workspace/*.json`
- [ ] Document current analyze artifact responsibilities in one internal reference doc if missing

### Acceptance Criteria

- Current analyze JSON outputs are covered by stable tests
- Current wiki output contract is covered by at least one integration test
- Current task routing still reads analyze outputs without behavioral drift
- There is a clear safety net for later refactors

### Out of Scope

- No behavior changes yet
- No new data model yet
- No new AI behavior yet

## Phase B: Add Evidence Layer and Knowledge Item IDs

**Theme:** Make knowledge traceable.

**Goal:** Introduce item-level knowledge identity, provenance, and lifecycle metadata without breaking current downstream consumers.

### Files likely involved

- New: `src/analyze/evidence-collector.ts`
- New: `src/analyze/knowledge-items.ts`
- New: `src/analyze/knowledge-lifecycle.ts`
- Modify: `src/analyze/types.ts`
- Modify: `src/analyze/emit-knowledge.ts`
- Modify: `src/runtime/wiki.ts`
- Modify: `.bbg/knowledge` README templates if needed

### Tasks

- [ ] Define `EvidenceItem`, `AmbiguitySignal`, and `AnalyzeKnowledgeItem` types
- [ ] Generate stable IDs for major knowledge items
- [ ] Add provenance fields and confidence metadata to emitted workspace knowledge
- [ ] Add lightweight comparison against previous analyze baseline
- [ ] Persist lifecycle metadata beside existing JSON knowledge outputs
- [ ] Ensure old task consumers continue working even if they ignore the new metadata

### Acceptance Criteria

- Every major analyze knowledge object has a stable ID
- Every major analyze knowledge object carries provenance and confidence metadata
- Previous-vs-current baseline comparison exists for changed knowledge items
- Existing task routing still works unchanged

### Risks

- Breaking JSON consumers by over-changing file shape
- Generating unstable IDs that churn between runs

### Guardrails

- Prefer additive schema changes over replacing current JSON structure
- Keep legacy fields until all runtime consumers migrate

**Theme:** Replace static gap filling with adaptive questioning.

**Goal:** Add the new evidence-driven interview engine behind a non-default flag so it can be tested safely against the current pipeline.

### Files likely involved

- New: `src/analyze/socratic-engine.ts`
- New: `src/analyze/interview-dimensions.ts`
- New: `src/analyze/question-generator.ts`
- New: `src/analyze/answer-processor.ts`
- Modify: `src/analyze/deep-interview.ts`
- Modify: `src/analyze/orchestrator.ts`
- Modify: `src/analyze/analysis-model.ts`

### Tasks

- [ ] Implement evidence-driven ambiguity detection
- [ ] Implement universal and project-adaptive interview dimensions
- [ ] Implement question generation from ambiguity signals
- [ ] Implement answer processing and score updates
- [ ] Support at least interactive and agent-assisted modes initially
- [ ] Add a feature flag such as internal `interview-v2` routing
- [ ] Preserve current interview path as fallback

### Acceptance Criteria

- The new engine can run end-to-end behind a flag
- Focus text influences ambiguity ranking and question priority
- The new engine feeds more than constraints/decisions/risks into the knowledge model
- The old interview path still works when the feature flag is disabled

### Out of Scope

- Full autonomous LLM self-interview can wait until a later sub-phase if needed

## Phase D: Ship Analyze UX 2.0

**Theme:** One command, AI-native execution.

**Goal:** Preserve `bbg analyze` as the only main entry point while making AI-context execution the preferred default.

### Files likely involved

- `src/cli.ts`
- `src/runtime/analyze-handoff.ts`
- `src/runtime/agent-tool.ts`
- `src/runtime/agent-runner.ts`
- UI copy / i18n files
- tests covering CLI behavior and handoff paths

### Tasks

- [ ] Keep `bbg analyze [focus...]` as the primary entry point
- [ ] Preserve focus as positional text rather than introducing `--focus`
- [ ] Add or improve auto-launch behavior when AI runner config is available
- [ ] Reduce terminal fallback messaging to one clear instruction path
- [ ] Persist the user's selected AI tool as default if appropriate
- [ ] Ensure the same command shape works for full analysis and focused analysis

### Acceptance Criteria

- User can still learn analyze with one command
- Running outside AI context prefers auto-launch over manual replay
- Running inside AI context executes directly
- Focused business analysis does not require extra commands or flags

### Risks

- Overcomplicating tool routing UX
- Exposing internal handoff mechanisms as user concepts

### Guardrails

- No new normal-path flags for AI mode, Hermes mode, or focus mode
- Keep manual replay only as fallback

## Phase E: Connect Analyze Outputs to Hermes Candidate Lifecycle

**Theme:** Turn analyze results into governed learning inputs.

**Goal:** Make selected analyze outputs first-class Hermes runtime artifacts and candidate sources.

### Files likely involved

- `src/analyze/orchestrator.ts`
- `src/analyze/emit-knowledge.ts`
- Hermes command/runtime support files
- Hermes schema templates if new metadata tables or fields are required
- Hermes asset tests and docs

### Tasks

- [ ] Register analyze runs and their knowledge snapshots as Hermes runtime artifacts
- [ ] Create Hermes candidates from selected new or changed analyze items
- [ ] Attach evidence links and recommended draft targets
- [ ] Expose analyze-origin candidates in Hermes review flows
- [ ] Document candidateworthiness rules by analyze item type

### Acceptance Criteria

- A successful analyze run can be inspected through Hermes as an artifact source
- High-value new or changed analyze items appear as candidate objects
- Candidate objects retain source run ID and evidence links
- No project-local speculative knowledge is promoted automatically

### Guardrails

- Candidate creation is selective, not blanket
- Promotion remains gated and human-reviewed

## Phase F: Add Task and Verification Back-Reference Loops

**Theme:** Let real work correct and strengthen analysis.

**Goal:** Allow tasks, verification, and incident-style workflows to confirm, refine, or contradict prior analyze knowledge items.

### Files likely involved

- `src/runtime/tasks.ts`
- verification runtime and wiki artifact writers
- Hermes runtime/evaluation integration points
- analyze comparison and lifecycle support files

### Tasks

- [ ] Allow runtime task context to carry matched knowledge item IDs
- [ ] Allow verify flows to emit `confirmed/refined/contradicted/unrelated` results for knowledge items
- [ ] Record validation events as Hermes evidence
- [ ] Apply confidence uplift or downgrade rules from task/verification outcomes
- [ ] Mark stale or contradicted knowledge items without silent overwrites

### Acceptance Criteria

- A verification result can point back to concrete analyze knowledge items
- Contradicted knowledge is visibly marked rather than silently replaced
- Confirmed knowledge gains confidence and lifecycle progress
- Future analyze runs can ingest prior runtime validation evidence

### Risks

- Knowledge corruption from noisy runtime signals
- Overlinking every task artifact to too many knowledge items

### Guardrails

- Only explicit matched IDs can change item state
- Confidence changes should be bounded and auditable

## Product Guardrails

Do **not** do the following during Analyze 2.0 rollout:

- auto-promote AI-inferred knowledge to global BBG assets
- force users to learn multiple analyze command variants
- require users to manage handoff files manually in the normal case
- break current `.bbg/knowledge/` consumers without a compatibility path
- treat every analyze item as candidate-worthy
- overwrite trusted knowledge without contradiction tracking
- couple project-local facts too early into global BBG memory

---

## Recommended Sequencing for Real Delivery

If implementation capacity is limited, the best sequence is:

1. **Phase A** — lock contracts
2. **Phase B** — add IDs and provenance
3. **Phase D** — improve UX and AI-first routing
4. **Phase C** — add the new Socratic engine behind a flag
5. **Phase E** — Hermes ingestion
6. **Phase F** — validation feedback loop
7. **Phase G** — default switch and cleanup

Why this order:

- A+B create the data substrate
- D fixes the user experience early
- C delivers analysis depth once the substrate exists
- E+F complete the knowledge loop
- G only happens after confidence is earned

---

## Success Definition

Analyze 2.0 is complete when all of the following are true:

- `bbg analyze` still feels like one simple command to the user
- deep analysis runs in AI context by default
- focus text can prioritize a business function without a new command family
- analyze outputs carry stable IDs, provenance, confidence, and lifecycle metadata
- future tasks automatically reuse analyze knowledge
- verification and runtime outcomes can strengthen or contradict prior knowledge
- Hermes can review analyze-origin candidates as part of the governed learning loop
- repeated analyze runs produce a stronger project baseline over time

In short:

> Analyze 2.0 succeeds when project understanding becomes durable, reusable, auditable, and progressively smarter, without making the command harder for users to adopt.

## 1. Goal

Define a concrete data model for Analyze 2.0 so that:

1. analyze output is stable and machine-consumable
2. knowledge items are identifiable across runs
3. provenance and confidence are explicit
4. Hermes can ingest analyze output without lossy translation
5. existing `.bbg/knowledge/` consumers can keep working during migration

This document answers the engineering question:

> What exact objects should Analyze 2.0 emit so the system can support evidence-driven analysis, lifecycle tracking, runtime reuse, and Hermes learning?

---

## 2. Design Principles

### 2.1 Additive First

Analyze 2.0 should extend current JSON outputs before replacing them.

Reason:

- `src/runtime/tasks.ts` already consumes current workspace JSON files directly
- Hermes query and wiki generation already depend on current file families
- a hard schema break would increase rollout risk unnecessarily

### 2.2 Stable Item Identity

Every meaningful knowledge object must have a stable ID that survives multiple analyze runs unless the underlying concept truly changes.

### 2.3 Item-Level Provenance

Document-level metadata is not enough. Provenance must exist at the knowledge-item level so future task and verification artifacts can confirm or contradict specific prior items.

### 2.4 Runtime JSON, Optional DB Index

The primary operational substrate remains JSON under `.bbg/knowledge/`. SQLite is a helper index and governance substrate, not the primary runtime content store.

### 2.5 Canonical Separation

The model must clearly separate:

- analyze runtime output
- canonical wiki pages
- Hermes candidate records
- promoted local/global knowledge

## 3. Current State Summary

Current analyze writes these families:

### Repo-level

- `.bbg/knowledge/repos/<repo>/technical.json`
- `.bbg/knowledge/repos/<repo>/business.json`
- `.bbg/knowledge/repos/<repo>/patterns.json`

### Workspace-level

- `topology.json`
- `integration-map.json`
- `business-modules.json`
- `capabilities.json`
- `analysis-dimensions.json`
- `critical-flows.json`
- `contracts.json`
- `domain-model.json`
- `business-context.json`
- `constraints.json`
- `risk-surface.json`
- `decisions.json`
- `change-impact.json`
- `focused-analysis.json` (optional)

Current issues:

1. items do not have stable IDs
2. provenance is file-level at best, not item-level
3. confidence exists on some nested objects but not consistently
4. no lifecycle state exists
5. Hermes cannot ingest selected items without custom transformation logic
6. comparison across runs is difficult

## 4. Target Storage Model

Analyze 2.0 should keep the current folder layout but add two new layers of structure.

### 4.1 Layer A: Current Domain Files Remain

Keep domain-oriented files for compatibility and direct runtime consumption.

Examples:

- `capabilities.json`
- `critical-flows.json`
- `contracts.json`

### 4.2 Layer B: Knowledge Item Index

Add a normalized item index that references all major analyze knowledge objects.

Recommended new files:

```text
.bbg/knowledge/workspace/knowledge-items.json
.bbg/knowledge/workspace/evidence-index.json
.bbg/knowledge/workspace/lifecycle.json
.bbg/knowledge/workspace/run-diff.json
```

### 4.3 Layer C: Analyze Run Snapshot

Persist a per-run summary that can be registered into Hermes.

Recommended path:

```text
.bbg/analyze/runs/<runId>/knowledge-snapshot.json
```

This becomes the bridge object from analyze to Hermes.

## 5. Core Objects

### 5.1 Analyze Knowledge Item

This is the core normalized object.

```typescript
interface AnalyzeKnowledgeItem {
  id: string;
  runId: string;
  scope: "workspace" | "repo";
  repo: string | null;
  kind:
    | "capability"
    | "critical-flow"
    | "contract-surface"
    | "domain-context"
    | "runtime-constraint"
    | "risk-item"
    | "decision-record"
    | "change-impact"
    | "analysis-dimension";
  title: string;
  summary: string;
  payloadPath: string;
  payloadPointer: string;
  confidence: number;
  status:
    | "observed"
    | "inferred"
    | "candidate"
    | "confirmed-local"
    | "promoted-local"
    | "eligible-global"
    | "promoted-global"
    | "stale"
    | "superseded";
  tags: string[];
  relatedIds: string[];
  provenance: KnowledgeProvenance[];
  freshness: KnowledgeFreshness;
  history: KnowledgeHistory;
}
```

### 5.2 Knowledge Provenance

```typescript
interface KnowledgeProvenance {
  source:
    | "static-analysis"
    | "business-signal"
    | "interview-answer"
    | "llm-inference"
    | "task-runtime"
    | "verification"
    | "incident-review"
    | "human-confirmation";
  ref: string;
  description: string;
  confidenceImpact: number;
  codeRefs: CodeReference[];
}

interface CodeReference {
  repo: string;
  file: string;
  lineRange: [number, number];
  symbolName?: string;
  snippet?: string;
}
```

### 5.3 Knowledge Freshness

```typescript
interface KnowledgeFreshness {
  createdAt: string;
  updatedAt: string;
  lastValidatedAt: string | null;
  lastSourceChangeAt: string | null;
  freshnessStatus: "fresh" | "needs-review" | "stale";
}
```

### 5.4 Knowledge History

```typescript
interface KnowledgeHistory {
  firstSeenRunId: string;
  lastSeenRunId: string;
  changeKind: "new" | "unchanged" | "strengthened" | "weakened" | "replaced";
  supersedes: string[];
  supersededBy: string | null;
}
```

## 6. Stable ID Rules

### 6.1 ID Requirements

IDs must:

- remain stable across runs when the same concept persists
- differ when materially distinct concepts appear
- not depend on timestamps
- be deterministic from semantic anchors

### 6.2 Recommended ID Strategy

```text
ak:<kind>:<workspace-or-repo-scope>:<semantic-slug>:<short-hash>
```

Examples:

```text
ak:capability:workspace:order-management:9f31a2
ak:critical-flow:workspace:payment-refund-flow:c7d120
ak:contract-surface:repo:shop-server:order-api:66ab90
ak:risk-item:workspace:inventory-double-deduction:43ce19
```

### 6.3 Semantic Anchor Rules by Kind

| Kind               | Recommended semantic anchor                       |
| ------------------ | ------------------------------------------------- |
| capability         | normalized capability name + owning repos         |
| critical-flow      | normalized flow summary + participating repos     |
| contract-surface   | normalized contract name + owners + boundary type |
| domain-context     | normalized context name + owner repo              |
| runtime-constraint | normalized statement + category                   |
| risk-item          | normalized title + affected repos                 |
| decision-record    | normalized statement                              |
| change-impact      | target + impacted repos                           |
| analysis-dimension | normalized dimension name                         |

### 6.4 ID Migration Rule

If a concept changes enough that the semantic anchor changes materially, emit a new ID and link the old one via `supersedes` / `supersededBy`.

## 7. Evidence Model

### 7.1 Evidence Item

Evidence must be separately indexable because one evidence item can support multiple knowledge items.

```typescript
interface EvidenceItem {
  id: string;
  runId: string;
  type:
    | "code-structure"
    | "route-entrypoint"
    | "api-entrypoint"
    | "domain-term"
    | "entity-term"
    | "integration-signal"
    | "risk-marker"
    | "interview-answer"
    | "llm-reasoning"
    | "verification-result"
    | "task-confirmation";
  summary: string;
  sourceRefs: string[];
  codeRefs: CodeReference[];
  clarity: number;
  relatedKnowledgeIds: string[];
  contradictions: string[];
}
```

### 7.2 Evidence Index File

Recommended file shape:

```typescript
interface AnalyzeEvidenceIndexFile {
  version: 2;
  updatedAt: string;
  runId: string;
  source: "analyze";
  evidence: EvidenceItem[];
}
```

### 7.3 Why a Separate Evidence Index

Benefits:

- prevents duplication across domain files
- gives question generation a consistent source
- makes contradiction tracking feasible
- gives Hermes candidate creation direct evidence references

## 8. Domain File Evolution

### 8.1 Compatibility Rule

Current domain files should keep their existing top-level arrays, but each nested item should become ID-bearing.

### 8.2 Example: `capabilities.json`

Current shape:

```json
{
  "version": 1,
  "updatedAt": "...",
  "source": "analyze",
  "capabilities": [
    {
      "name": "Order management",
      "description": "...",
      "confidence": 0.82
    }
  ]
}
```

Target shape:

```json
{
  "version": 2,
  "updatedAt": "...",
  "runId": "an_2026_04_20_001",
  "source": "analyze",
  "capabilities": [
    {
      "id": "ak:capability:workspace:order-management:9f31a2",
      "name": "Order management",
      "description": "...",
      "owningRepos": ["shop-server"],
      "responsibilities": ["..."],
      "evidence": {
        "summary": "...",
        "signals": ["..."],
        "evidenceIds": ["ev:route:...", "ev:api:..."]
      },
      "confidence": 0.82,
      "status": "inferred",
      "provenance": [
        {
          "source": "business-signal",
          "ref": "repo-business-signals:shop-server",
          "description": "Derived from route and API signals",
          "confidenceImpact": 0.18,
          "codeRefs": []
        }
      ]
    }
  ]
}
```

### 8.3 Apply the Same Pattern To

- `critical-flows.json`
- `contracts.json`
- `domain-model.json`
- `risk-surface.json`
- `decisions.json`
- `change-impact.json`
- `analysis-dimensions.json`

Each nested item should gain:

- `id`
- `status`
- `provenance`
- `evidence.evidenceIds` where applicable

## 9. Normalized Workspace Index Files

### 9.1 `knowledge-items.json`

Recommended shape:

```typescript
interface AnalyzeKnowledgeItemsFile {
  version: 2;
  updatedAt: string;
  runId: string;
  source: "analyze";
  items: AnalyzeKnowledgeItem[];
}
```

Purpose:

- one-stop normalized item registry
- easy Hermes ingestion
- easy run-to-run diffing
- easy runtime matching by ID

### 9.2 `lifecycle.json`

Recommended shape:

```typescript
interface AnalyzeLifecycleFile {
  version: 2;
  updatedAt: string;
  runId: string;
  source: "analyze";
  states: Array<{
    knowledgeItemId: string;
    status: AnalyzeKnowledgeItem["status"];
    confidence: number;
    freshnessStatus: KnowledgeFreshness["freshnessStatus"];
    reviewRequired: boolean;
    supersededBy: string | null;
  }>;
}
```

Purpose:

- cheap status scan without loading all domain payloads
- easy stale/review workflow integration

### 9.3 `run-diff.json`

Recommended shape:

```typescript
interface AnalyzeRunDiffFile {
  version: 2;
  updatedAt: string;
  runId: string;
  previousRunId: string | null;
  source: "analyze";
  summary: {
    newItems: number;
    unchangedItems: number;
    strengthenedItems: number;
    weakenedItems: number;
    supersededItems: number;
    removedItems: number;
  };
  changes: Array<{
    knowledgeItemId: string;
    changeKind: KnowledgeHistory["changeKind"] | "removed";
    previousConfidence: number | null;
    currentConfidence: number | null;
    note: string;
  }>;
}
```

Purpose:

- Hermes candidate selection
- wiki refresh targeting
- human review focus

## 11. Hermes Mapping

### 11.1 Mapping Strategy

Hermes should not parse raw domain files ad hoc. It should read the normalized analyze item index and snapshot.

### 11.2 Mapping Table

| Analyze object                                      | Hermes destination                |
| --------------------------------------------------- | --------------------------------- |
| `knowledge-snapshot.json`                           | `hermes_run_artifacts`            |
| `AnalyzeKnowledgeItem` with candidate-worthy change | `hermes_candidates`               |
| `provenance` and `evidenceIds`                      | `hermes_candidate_evidence`       |
| `run-diff.json`                                     | candidate selection input         |
| `lifecycle.json`                                    | verification and review targeting |

### 11.3 Candidate Metadata Extension

Recommended analyze-origin candidate payload fields:

```typescript
interface AnalyzeOriginCandidateMeta {
  sourceKind: AnalyzeKnowledgeItem["kind"];
  sourceKnowledgeItemId: string;
  sourceRunId: string;
  recommendedTarget: "wiki" | "process" | "rule" | "skill" | "report";
  changeKind: KnowledgeHistory["changeKind"];
  confidence: number;
}
```

### 12.1 Runtime Matching Contract

Task routing should be able to return both human-readable guidance and the underlying matched IDs.

```typescript
interface TaskImpactGuidance {

  matchedCapabilities: string[];
  matchedFlows: string[];

  matchedRisks: string[];

  impactedRepos: string[];
  reviewerHints: string[];
  rationale: string[];
}
```

### 12.2 Verification Back-Reference Contract

```typescript
interface KnowledgeValidationEvent {
  id: string;
  knowledgeItemId: string;
  source: "verify" | "task-runtime" | "incident-review" | "human-review";
  runId: string | null;
  recordedAt: string;
  outcome: "confirmed" | "refined" | "contradicted" | "unrelated";
  confidenceDelta: number;
  notes: string;

```

These events can live in Hermes tables or a JSON event log, but the object contract should be explicit either way.

## 13. Versioning Strategy

### 13.1 File Versions

Recommended version policy:

- existing analyze knowledge files move from `version: 1` to `version: 2` when IDs/provenance/status are added
- new normalized files start at `version: 2`
- per-run snapshot starts at `version: 1`

### 13.2 Compatibility Policy

For one migration window:

1. existing consumers may ignore new fields
2. new files are additive and optional for old consumers
3. old domain file array names remain unchanged
4. task routing migration should happen before removing any legacy fields

### 13.3 Breaking Change Rule

Do not rename current top-level keys like:

- `capabilities`
- `flows`
- `contractSurfaces`
- `contexts`
- `risks`
- `decisions`
- `impacts`

until all consumers are migrated.

## 15. Migration Plan

### Step 1

Add IDs, provenance, and status fields into existing domain files.

### Step 2

Emit `knowledge-items.json` and `evidence-index.json`.

### Step 3

Emit `lifecycle.json` and `run-diff.json`.

### Step 4

Update Hermes ingestion and runtime consumers to prefer normalized indexes.

### Step 5

Only after all consumers migrate, consider pruning any redundant legacy fields.

## 16. Anti-Patterns

1. Replacing current domain files with only one giant normalized blob
2. Using unstable IDs derived from timestamps or array order
3. Tracking provenance only at the file level
4. Storing all analyze output only in SQLite and removing JSON runtime files
5. Making Hermes parse raw domain documents without a normalized item registry
6. Silently deleting replaced knowledge items instead of marking supersession
7. Designing the model around one specific LLM provider

---

## 17. Final Recommendation

The recommended Analyze 2.0 data model is:

- keep current `.bbg/knowledge/workspace/*.json` domain files for compatibility
- enrich nested objects with `id`, `status`, `provenance`, and explicit evidence links
- add normalized indexes for knowledge items, evidence, lifecycle state, and run diffs
- add a per-run snapshot specifically for Hermes registration
- let JSON remain the primary runtime substrate, with Hermes/SQLite acting as governed indexing and lifecycle infrastructure

In short:

> Analyze 2.0 should not throw away the current file model. It should normalize and enrich it so that project knowledge becomes identifiable, evidence-backed, comparable across runs, and reusable by both runtime workflows and Hermes.

# BBG Harness 9.5 设计规格

**创建时间**: 2026-04-03
**状态**: Approved (设计阶段)
**作者**: AI + 人工协作
**范围**: 从 Harness 8/10 提升到 9.5/10 + Agent MBTI 人格化
**前置文档**: `docs/maintenance/specs/2026-04-02-bbg-harness-engineering-prd.md`

---

## 一、总览

### 1.1 目标

通过 8 个能力模块的设计和实施，将 BBG 生成到目标项目的 AI 开发 Harness 从 8/10 完成度提升到 9.5/10，同时为全部 25 个 Agent 添加 MBTI 深度人格建模。

### 1.2 核心原则

**BBG = 生成器/安装器**。所有新能力都是 BBG 生成到目标项目中的治理资产，而不是 BBG 自身使用的功能。

- BBG 的 `src/` 代码负责分析目标项目、渲染模板、安装内容
- 目标项目的 `.bbg/` 目录承载运行时状态
- 目标项目的 `skills/`、`commands/`、`hooks/` 承载治理行为

### 1.3 设计决策汇总

| 决策项             | 选择                                          | 理由                                 |
| ------------------ | --------------------------------------------- | ------------------------------------ |
| 设计范围           | 8 个模块全覆盖，实施分期                      | 设计阶段覆盖全貌，实施阶段逐步交付   |
| MBTI 深度          | 深度人格建模（行为指令化）                    | 仅标注 MBTI 类型对 AI 行为影响有限   |
| 组织级治理         | 预留架构，本次不实现                          | 核心用户场景仍是项目级               |
| 数据存储           | SQLite（生成到目标项目）                      | telemetry 趋势分析需要聚合查询能力   |
| 运行时编排         | 轻量编排器（调度+状态，执行委托 AI 工具）     | BBG 核心价值是治理而非替代 AI 工具   |
| Repo Map           | 文件级通用 + 5 语言符号级                     | 文件级作为通用基线，符号级按语言扩展 |
| 需求挖掘           | 苏格拉底式模糊度评分 + 结构化提问             | 在设计之前把需求理解清楚             |
| Markdown vs SQLite | 共存：Markdown 存内容，SQLite 存元数据        | 各取所长                             |
| 红队测试           | 2 轮协议（系统化 + 对抗性），后端项目条件生成 | 后端 API 安全风险高，需结构化覆盖    |

### 1.4 实施阶段（方案 A：按能力层垂直切分）

```
Phase 1: Agent MBTI 人格化（纯内容变更，风险最低，立即可见效果）
Phase 2: 观测/Telemetry（SQLite schema + 模板 + 报表生成）
Phase 3: Eval/Benchmark（黄金任务集 + 回归对比框架）
Phase 4: 强策略执行（风险分级 + 审批流 + 权限边界）
Phase 5: Context Engineering（repo map + symbol map + 上下文预算）
Phase 6: 运行时编排 + 红队测试（workflow runner + 后端红队 2 轮协议）
Phase 7: 苏格拉底式需求挖掘（deep-interview skill）
Phase 8: 组织级治理（预留 schema + 接口定义，不实现）
```

## 二、Agent MBTI 人格化

### 2.1 设计目标

将 MBTI 性格特征转化为可执行的行为指令，让 AI 工具在扮演不同 Agent 角色时展现明显的人格差异。

### 2.2 Agent 文件结构变更

在现有 YAML frontmatter 中新增 `personality` 字段块：

```yaml
---
name: architect
description: System design and architecture decisions
tools: ["Read", "Grep", "Glob"]
model: opus
personality:
  mbti: INTJ
  label: "战略架构师"
  traits:
    - 系统性思维，先全局后局部
    - 偏好长期可维护性而非短期便利
    - 决策时依赖逻辑推演而非直觉
  communication:
    style: 结构化、精确、有层次
    tendency: 先给结论再展开论证
    weakness: 可能忽略实现者的感受，需要刻意关注可落地性
---
```

Markdown body 中的开头段落升级为人格化角色定义：

```markdown
# Architect

You are a strategic systems thinker (INTJ) who sees architecture as
long-term investment. You naturally gravitate toward structured analysis
— mapping dependencies, evaluating trade-offs, and designing for
maintainability over convenience. Present your reasoning in layers:
conclusion first, then supporting analysis, then alternatives considered.
When recommending, be direct about your preference and why, but
acknowledge practical constraints that implementers will face.
```

```sql
-- 核心事件表
CREATE TABLE IF NOT EXISTS telemetry_events (
  id          INTEGER PRIMARY KEY AUTOINCREMENT,
  timestamp   TEXT    NOT NULL DEFAULT (datetime('now')),
  session_id  TEXT,
  event_type  TEXT    NOT NULL,  -- 'command_run' | 'hook_trigger' | 'gate_result' | 'model_call' | 'workflow_step'

  id          TEXT    PRIMARY KEY,

  ended_at    TEXT,
```

#### 视图（6 个）

```sql
CREATE VIEW IF NOT EXISTS v_daily_gate_pass_rate AS
SELECT date(timestamp) AS day, check_name,
  COUNT(*) AS total, SUM(passed) AS passed,
  ROUND(100.0 * SUM(passed) / COUNT(*), 1) AS pass_rate
FROM gate_results GROUP BY day, check_name;

CREATE VIEW IF NOT EXISTS v_hook_effectiveness AS
SELECT hook_name, COUNT(*) AS total_triggers,
  SUM(CASE WHEN action = 'blocked' THEN 1 ELSE 0 END) AS blocks,
  SUM(false_positive) AS false_positives,
  ROUND(100.0 * SUM(false_positive) / NULLIF(COUNT(*), 0), 1) AS false_positive_rate
FROM hook_hits GROUP BY hook_name;

CREATE VIEW IF NOT EXISTS v_session_summary AS
SELECT date(started_at) AS day, tool,
  COUNT(*) AS sessions, AVG(commands) AS avg_commands, SUM(commands) AS total_commands
FROM sessions GROUP BY day, tool;

CREATE VIEW IF NOT EXISTS v_task_flow_efficiency AS
SELECT task_id, title,
  MIN(CASE WHEN phase = 'requirement' THEN timestamp END) AS started_at,
  MIN(CASE WHEN phase = 'completed' THEN timestamp END) AS completed_at,
  COUNT(DISTINCT phase) AS phases_touched,
  ROUND(julianday(MIN(CASE WHEN phase = 'completed' THEN timestamp END))
    - julianday(MIN(CASE WHEN phase = 'requirement' THEN timestamp END)), 2) AS days_to_complete
FROM task_lifecycle GROUP BY task_id, title;

CREATE VIEW IF NOT EXISTS v_tech_evolution_timeline AS
SELECT timestamp, decision_id, title, category, status, supersedes,
  CASE WHEN supersedes IS NOT NULL THEN '→ 替代 ' || supersedes ELSE '新决策' END AS evolution_note
FROM decision_records ORDER BY timestamp;

CREATE VIEW IF NOT EXISTS v_model_route_efficiency AS
SELECT task_type, model,
  COUNT(*) AS total_calls, AVG(tokens_in + tokens_out) AS avg_tokens,
  ROUND(100.0 * SUM(CASE WHEN status = 'success' THEN 1 ELSE 0 END) / COUNT(*), 1) AS success_rate,
  AVG(duration_ms) AS avg_duration_ms
FROM model_routes GROUP BY task_type, model;
```

### 3.6 兼容策略

- `.bbg/telemetry/events.json` 保留不删除
- SQLite 是可选增强，由 `telemetry-init.sql` 初始化
- **初始化触发方式**：`session-start.js` hook 在会话启动时检测 `telemetry.db` 是否存在，不存在则执行 `telemetry-init.sql` 创建。后续模块的 SQL schema 也在此时一并初始化（幂等的 `CREATE TABLE IF NOT EXISTS`）。
- `telemetry-dashboard` skill 优先从 SQLite 查询，fallback 到 JSON
- `.bbg/telemetry.db` 加入目标项目的 `.gitignore`

### 3.7 Markdown vs SQLite 分工

- **保留** Markdown 作为内容载体（spec、plan、ADR 正文仍是 `.md` 文件）
- **消除** 纯粹用于追踪状态的重复 Markdown 元数据（转移到 SQLite）
- **SQLite 记录引用路径**（`spec_path`、`plan_path`），建立文件与结构化数据的关联

模式：AI 工具产出 spec/plan/ADR 时，先写 markdown 文件，再向 SQLite 注册一条记录。

### 4.3 SQLite Eval 表

```sql
CREATE TABLE IF NOT EXISTS eval_runs (
  id            INTEGER PRIMARY KEY AUTOINCREMENT,
  timestamp     TEXT    NOT NULL DEFAULT (datetime('now')),
  run_id        TEXT    NOT NULL UNIQUE,
  harness_version TEXT,
  task_set      TEXT    NOT NULL,
  model         TEXT,
  total_tasks   INTEGER NOT NULL,
  passed        INTEGER NOT NULL,
  failed        INTEGER NOT NULL,
  skipped       INTEGER DEFAULT 0,
  duration_ms   INTEGER,
  config_snapshot TEXT
);

CREATE TABLE IF NOT EXISTS eval_task_results (
  id            INTEGER PRIMARY KEY AUTOINCREMENT,
  run_id        TEXT    NOT NULL REFERENCES eval_runs(run_id),
  task_id       TEXT    NOT NULL,
  task_title    TEXT,
  category      TEXT,
  status        TEXT    NOT NULL,  -- 'pass' | 'fail' | 'partial' | 'skip' | 'error'
  score         REAL,
  duration_ms   INTEGER,
  tokens_used   INTEGER,
  iterations    INTEGER,
  failure_reason TEXT,
  grader_output TEXT
);

CREATE VIEW IF NOT EXISTS v_eval_version_compare AS
SELECT r.harness_version, r.model, r.task_set,
  COUNT(t.id) AS total_tasks,
  ROUND(100.0 * SUM(CASE WHEN t.status = 'pass' THEN 1 ELSE 0 END) / COUNT(t.id), 1) AS pass_rate,
  ROUND(AVG(t.score) * 100, 1) AS avg_score,
  SUM(t.tokens_used) AS total_tokens,
  AVG(t.iterations) AS avg_iterations
FROM eval_runs r JOIN eval_task_results t ON r.run_id = t.run_id
GROUP BY r.harness_version, r.model, r.task_set;
```

### 4.4 Golden Task Set

`manifest.json` 定义任务集，BBG 生成 4 个 starter 黄金任务（按 Handlebars 模板根据目标项目语言渲染）：

| 任务             | 类别     | Workflow       | 评估维度               |
| ---------------- | -------- | -------------- | ---------------------- |
| simple-bugfix    | bugfix   | tdd            | 能否遵循 TDD 修复 bug  |
| tdd-feature      | feature  | tdd + plan     | 能否先写测试再实现     |
| security-review  | review   | security-scan  | 能否发现已知安全漏洞   |
| refactor-extract | refactor | refactor-clean | 能否识别并提取重复代码 |

### 4.5 回归对比流程

1. 运行当前 Harness 版本的 golden task set → 结果入库
2. 查询上一个 baseline 的结果
3. 对比 pass_rate、avg_score、tokens、iterations
4. 产出回归报告

**策略配置为 Handlebars 模板**（模板文件 `policy.json.hbs`，BBG 渲染后部署为目标项目的 `.bbg/policy/policy.json`），不同类型项目有不同默认策略：

- 前端项目：`public/` 和 `next.config.*` 为 medium risk

### 5.4 风险评估流程

```
操作请求 → 路径匹配 → 操作分类 → 风险定级 → 策略决策
                                                  ↓
                                    allow → 直接执行
                                    warn → 输出警告 + 执行 + 记录
                                    require-approval → 暂停等待人工确认 + 记录
                                    block → 拒绝执行 + 记录
```

### 7.3 Workflow 定义格式

> **注意**：`success_criteria` 是给 AI 工具的自然语言指令，由 AI 工具自行评估是否满足，不是自动检测条件。Workflow runner 仅跟踪状态转换，不执行评判。

```yaml

title: "TDD 新功能开发"
version: "1.0"
config:
  max_retries: 2
  retry_delay_ms: 5000
  timeout_ms: 3600000
  allow_skip: false
steps:
  - id: plan
    title: "需求分析与规划"
    agent: planner
    command: plan
    outputs: ["plan_path"]
    success_criteria:
      - "产出实施计划文件"
  - id: write-test
    title: "编写失败测试 (RED)"
    agent: tdd-guide
    command: tdd
    depends_on: [plan]

    outputs: ["test_file_path"]
    success_criteria:

    retry:
      on: ["test_not_failing"]
      max: 2
  - id: implement
    title: "最小实现 (GREEN)"
    agent: tdd-guide
    depends_on: [write-test]
    success_criteria:
      - "所有测试通过"
  - id: refactor
    title: "重构优化 (IMPROVE)"
    agent: refactor-cleaner
    depends_on: [implement]
  - id: review
    title: "代码评审"
    agent: code-reviewer
    command: code-review
    depends_on: [refactor]
  - id: verify
    title: "最终验证"
    agent: e2e-runner
    command: verify
    depends_on: [review]

  notify: true
on_complete:
  - "记录到 telemetry"
  - "创建 checkpoint"
```

### 7.4 状态机

```
步骤: pending → in_progress → completed
                    ↓
                failed → retrying → in_progress
                    ↓
                paused → in_progress (resume)
                    ↓
                aborted

Workflow 整体: pending → in_progress → completed / aborted
                              ↓
                          paused (等待人工介入)
```

### 7.5 SQLite 编排表

```sql
CREATE TABLE IF NOT EXISTS workflow_instances (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  workflow_id TEXT NOT NULL UNIQUE, definition TEXT NOT NULL,
  task_id TEXT, started_at TEXT NOT NULL, completed_at TEXT,
  status TEXT NOT NULL, current_step TEXT, session_id TEXT,
  total_steps INTEGER, completed_steps INTEGER DEFAULT 0,
  total_duration_ms INTEGER
);

CREATE TABLE IF NOT EXISTS workflow_steps (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  workflow_id TEXT NOT NULL REFERENCES workflow_instances(workflow_id),
  step_id TEXT NOT NULL, agent TEXT, status TEXT NOT NULL,
  started_at TEXT, completed_at TEXT, duration_ms INTEGER,
  retries INTEGER DEFAULT 0, failure_reason TEXT, outputs TEXT
);

CREATE VIEW IF NOT EXISTS v_workflow_efficiency AS
SELECT wi.definition, COUNT(*) AS total_runs,
  SUM(CASE WHEN wi.status = 'completed' THEN 1 ELSE 0 END) AS completed,
  ROUND(100.0 * SUM(CASE WHEN wi.status = 'completed' THEN 1 ELSE 0 END) / COUNT(*), 1) AS completion_rate,
  AVG(wi.total_duration_ms) AS avg_duration_ms
FROM workflow_instances wi GROUP BY wi.definition;

CREATE VIEW IF NOT EXISTS v_step_bottlenecks AS
SELECT wi.definition, ws.step_id, ws.agent,
  COUNT(*) AS executions, AVG(ws.duration_ms) AS avg_duration_ms,
  SUM(ws.retries) AS total_retries
FROM workflow_steps ws JOIN workflow_instances wi ON ws.workflow_id = wi.workflow_id
GROUP BY wi.definition, ws.step_id, ws.agent ORDER BY avg_duration_ms DESC;
```

### 7A.4 攻击分类学（基于 OWASP API Security Top 10 2023 + WSTG）

6 大攻击域，22 个攻击类别：

#### 域 1：认证与授权（Authentication & Authorization）

| ID    | 攻击类别              | OWASP 映射 | 严重性 | 描述                               |
| ----- | --------------------- | ---------- | ------ | ---------------------------------- |
| AA-01 | 对象级授权绕过 (BOLA) | API1:2023  | 严重   | 通过操纵资源 ID 访问其他用户的对象 |
| AA-02 | 功能级授权绕过 (BFLA) | API5:2023  | 严重   | 普通用户访问管理员级 API 端点      |
| AA-03 | 认证机制绕过          | API2:2023  | 严重   | 弱密码策略、JWT 缺陷、Token 泄露   |
| AA-04 | 对象属性级授权绕过    | API3:2023  | 高     | Mass Assignment、过度数据暴露      |

#### 域 2：输入验证与注入（Input Validation & Injection）

| ID    | 攻击类别               | OWASP 映射      | 严重性 | 描述                                                 |
| ----- | ---------------------- | --------------- | ------ | ---------------------------------------------------- |
| IV-01 | SQL / NoSQL / ORM 注入 | WSTG-INPV-05    | 严重   | 通过构造输入操纵数据库查询                           |
| IV-02 | 命令注入               | WSTG-INPV-12    | 严重   | 用户输入被拼接到 OS 命令中                           |
| IV-03 | SSRF                   | API7:2023       | 高     | 服务端被诱导向内部服务发起请求                       |
| IV-04 | SSTI                   | WSTG-INPV-18    | 高     | 模板引擎注入导致 RCE                                 |
| IV-05 | XSS / Header 注入      | WSTG-INPV-01/07 | 中     | 后端返回未转义的用户输入（API 响应/错误消息/Header） |

#### 域 3：业务逻辑（Business Logic）

| ID    | 攻击类别       | OWASP 映射   | 严重性 | 描述                                 |
| ----- | -------------- | ------------ | ------ | ------------------------------------ |
| BL-01 | 业务流程绕过   | API6:2023    | 高     | 跳过必要步骤、重放交易、绕过限制规则 |
| BL-02 | 竞态条件       | WSTG-BUSL-07 | 高     | 并发请求导致余额/库存/状态不一致     |
| BL-03 | 数据一致性破坏 | WSTG-BUSL-09 | 中     | 利用 API 调用顺序破坏业务不变量      |

#### 域 4：资源与配置（Resource & Configuration）

| ID    | 攻击类别          | OWASP 映射   | 严重性 | 描述                                   |
| ----- | ----------------- | ------------ | ------ | -------------------------------------- |
| RC-01 | 资源耗尽 (DoS)    | API4:2023    | 高     | 无限分页、深度嵌套 GraphQL、大文件上传 |
| RC-02 | 安全配置缺陷      | API8:2023    | 中     | 调试端点暴露、默认凭据、CORS 过宽      |
| RC-03 | API 版本/文档暴露 | API9:2023    | 中     | 旧版 API 未下线、Swagger 生产环境暴露  |
| RC-04 | 错误信息泄露      | WSTG-ERRH-01 | 低     | Stack trace、数据库名、内部路径泄露    |

#### 域 5：数据安全（Data Security）

| ID    | 攻击类别           | OWASP 映射 | 严重性 | 描述                              |
| ----- | ------------------ | ---------- | ------ | --------------------------------- |
| DS-01 | 敏感数据过度暴露   | API3:2023  | 高     | API 返回超出需要的用户/系统数据   |
| DS-02 | 弱加密 / 明文传输  | WSTG-CRYP  | 高     | HTTP 传输、弱哈希算法、密钥硬编码 |

| ----- | ---------------- | ------------ | ------ | ------------------------------------------ |

### 7A.5 两轮红队协议

#### Round 1：系统化扫描（Systematic Sweep）

**目标**：按攻击分类学逐项检查，确保无遗漏。

**执行方式**：AI 工具按照 `backend-red-team-playbook.md` 的检查清单，逐域逐类别进行代码审查和攻击尝试。

```
执行流程:
1. 收集 API 端点清单（从 router/controller 文件提取）
2. 对每个端点标注涉及的攻击域
3. 按 6 大域 × 22 类别逐项检查：
   a. 代码层面：是否存在该类型漏洞的代码模式
   b. 配置层面：是否存在不安全配置
   c. 构造攻击用例：给出具体的 payload / 请求构造
4. 每个发现记录：攻击ID、端点、payload、预期结果、实际结果、严重性
5. 产出 Round 1 报告
```

**通过标准**：所有「严重」和「高」级别发现已修复或有明确的缓解方案。

#### Round 2：对抗性创意攻击（Adversarial Creative Attack）

**目标**：模拟真实攻击者的思维，发现 Round 1 系统化扫描无法覆盖的组合攻击和逻辑漏洞。

**执行方式**：AI 工具以 `security-reviewer` Agent 人格（ISTP 冷静渗透者）进行创意攻击。

```
执行流程:
1. 审阅 Round 1 报告，了解已知攻击面
2. 识别高价值目标（支付、权限变更、数据导出、状态转换等关键业务流）
3. 设计攻击链（不是单点攻击，而是多步组合）：
   a. 链式攻击：A 漏洞的输出作为 B 漏洞的输入
   b. 时序攻击：利用并发/竞态创造异常状态
   c. 边界探测：极端值、编码混淆、Unicode 技巧
   d. 业务逻辑滥用：合法 API 的异常组合顺序
   e. 信任边界穿越：从低权限区域向高权限区域渗透
4. 对每个攻击链记录：攻击路径、前置条件、各步骤 payload、影响范围
5. 产出 Round 2 报告
```

**通过标准**：所有被发现的攻击链已有缓解方案，或已评估为可接受风险并记录理由。

#### 轮次判定与迭代

```
Round 1 → 修复 → Round 2 → 修复 → 最终判定
                                      ↓
                              all_critical_fixed = true → PASS
                              any_critical_open = true  → BLOCK（不可发布）
                              high_open > 0             → CONDITIONAL（需人工确认）
```

### 7A.6 严重性评分

采用简化的 CVSS 风格评分：

| 严重性 | 分值范围 | 处置策略                           |
| ------ | -------- | ---------------------------------- |
| 严重   | 9.0-10.0 | 必须修复，阻断发布                 |
| 高     | 7.0-8.9  | 应当修复，可附缓解方案后有条件发布 |
| 中     | 4.0-6.9  | 建议修复，记录风险接受理由         |
| 低     | 0.1-3.9  | 可选修复，记录即可                 |
| 信息   | 0.0      | 仅记录，无需处置                   |

```sql
CREATE TABLE IF NOT EXISTS red_team_rounds (
  id              INTEGER PRIMARY KEY AUTOINCREMENT,
  timestamp       TEXT    NOT NULL DEFAULT (datetime('now')),
  session_id      TEXT,
  workflow_id     TEXT,
  round_number    INTEGER NOT NULL,  -- 1 or 2
  round_type      TEXT    NOT NULL,  -- 'systematic' | 'adversarial'
  endpoints_checked INTEGER,
  categories_checked INTEGER,
  findings_total  INTEGER DEFAULT 0,
  findings_critical INTEGER DEFAULT 0,
  findings_high   INTEGER DEFAULT 0,
  findings_medium INTEGER DEFAULT 0,
  findings_low    INTEGER DEFAULT 0,
  verdict         TEXT,              -- 'PASS' | 'CONDITIONAL' | 'BLOCK'
  report_path     TEXT,
  duration_ms     INTEGER
);

CREATE TABLE IF NOT EXISTS red_team_findings (
  id              INTEGER PRIMARY KEY AUTOINCREMENT,
  round_id        INTEGER NOT NULL REFERENCES red_team_rounds(id),
  finding_id      TEXT    NOT NULL,  -- e.g., 'F-001'
  attack_id       TEXT    NOT NULL,  -- e.g., 'AA-01'
  domain          TEXT    NOT NULL,  -- e.g., 'auth_authz'
  endpoint        TEXT,
  severity        TEXT    NOT NULL,  -- 'critical' | 'high' | 'medium' | 'low' | 'info'
  score           REAL,
  title           TEXT    NOT NULL,
  status          TEXT    NOT NULL,  -- 'open' | 'mitigated' | 'fixed' | 'accepted'
  fixed_at        TEXT,
  details         TEXT               -- JSON: { description, payload, impact, remediation }
);

CREATE TABLE IF NOT EXISTS red_team_chains (
  id              INTEGER PRIMARY KEY AUTOINCREMENT,
  round_id        INTEGER NOT NULL REFERENCES red_team_rounds(id),
  chain_id        TEXT    NOT NULL,  -- e.g., 'C-001'
  title           TEXT    NOT NULL,
  steps           TEXT    NOT NULL,  -- JSON array of finding_ids
  preconditions   TEXT,
  impact          TEXT,
  mitigation      TEXT,
  status          TEXT    NOT NULL   -- 'open' | 'mitigated' | 'fixed'
);

CREATE VIEW IF NOT EXISTS v_red_team_trend AS
SELECT date(timestamp) AS day, round_type,
  COUNT(*) AS total_rounds,
  AVG(findings_critical + findings_high) AS avg_critical_high,
  SUM(CASE WHEN verdict = 'PASS' THEN 1 ELSE 0 END) AS pass_count,
  SUM(CASE WHEN verdict = 'BLOCK' THEN 1 ELSE 0 END) AS block_count
FROM red_team_rounds GROUP BY day, round_type;

CREATE VIEW IF NOT EXISTS v_attack_domain_heatmap AS
SELECT domain, severity, COUNT(*) AS finding_count,
  SUM(CASE WHEN status IN ('open', 'mitigated') THEN 1 ELSE 0 END) AS unresolved
FROM red_team_findings GROUP BY domain, severity ORDER BY finding_count DESC;
```

### 8.3 模糊度评分模型

**8 个评分维度**：

| 维度       | 权重 | 含义                  |
| ---------- | ---- | --------------------- |
| 意图清晰度 | 0.20 | 为什么要做这件事      |
| 预期结果   | 0.15 | 完成后是什么状态      |
| 范围边界   | 0.15 | 做到哪里停            |
| 约束条件   | 0.10 | 技术/业务限制         |
| 验收标准   | 0.15 | 怎么判断完成          |
| 非目标     | 0.10 | 明确不做什么          |
| 决策边界   | 0.10 | AI 可以自主决定什么   |
| 上下文理解 | 0.05 | 对现有代码/系统的理解 |

**公式**: `ambiguity = 1 - Σ(dimension_score × weight)`

**深度档位**：

| 档位            | 阈值   | 最大轮数 |
| --------------- | ------ | -------- |
| Quick           | ≤ 0.30 | 5        |
| Standard (默认) | ≤ 0.20 | 12       |
| Deep            | ≤ 0.15 | 20       |

### 8.4 提问流程

**Phase 1: 预检上下文** — AI 先自主探索（repo-map、symbol-map、历史决策），能从代码库获得的信息不问用户。

**Phase 2: 苏格拉底式提问循环** — 每轮：确定最弱维度 → 选择分析透镜 → 生成一个问题 → 等待回答 → 压力测试 → 重新评分 → 展示进度。

阶段优先级：

1. 意图优先：意图、预期结果、范围、非目标
2. 可行性：约束、验收标准、决策边界
3. 落地：上下文理解、技术细节

压力追问策略（不用"为什么"直接追问）：

1. 要求具体例子或反例
2. 暴露隐藏假设
3. 逼迫边界决策
4. 回溯本质

**Phase 3: 分析透镜** — 在波次间歇应用 2-3 个透镜：

- 战略透镜：负空间、机会成本、信心分级
- 系统透镜：依赖链、时间冲突、最小版本
- 挑战透镜：事前验尸、反转、终止条件

**Phase 4: 结晶产出** — 访谈记录 + 需求规格（含已验证/未验证假设、压力测试发现）

**Phase 5: 执行桥接** — 推荐 brainstorming → writing-plans → 实现

### 8.5 SQLite 访谈表

```sql
CREATE TABLE IF NOT EXISTS interview_sessions (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  interview_id TEXT NOT NULL UNIQUE, slug TEXT NOT NULL,
  profile TEXT NOT NULL, topic TEXT NOT NULL,
  started_at TEXT NOT NULL, completed_at TEXT,
  status TEXT NOT NULL, rounds INTEGER DEFAULT 0,
  initial_ambiguity REAL DEFAULT 1.0, final_ambiguity REAL,
  threshold REAL NOT NULL, spec_path TEXT, transcript_path TEXT
);

CREATE TABLE IF NOT EXISTS interview_rounds (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  interview_id TEXT NOT NULL REFERENCES interview_sessions(interview_id),
  round_num INTEGER NOT NULL, target_dimension TEXT NOT NULL,
  lens_used TEXT, question TEXT NOT NULL, answer TEXT,
  ambiguity_before REAL, ambiguity_after REAL,
  dimension_scores TEXT, pressure_type TEXT
);

CREATE TABLE IF NOT EXISTS interview_assumptions (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  interview_id TEXT NOT NULL, assumption TEXT NOT NULL,
  status TEXT NOT NULL, verified_at TEXT, evidence TEXT
);

CREATE VIEW IF NOT EXISTS v_interview_effectiveness AS
SELECT profile, COUNT(*) AS total_sessions,
  AVG(rounds) AS avg_rounds,
  AVG(initial_ambiguity - final_ambiguity) AS avg_ambiguity_reduction,
  SUM(CASE WHEN status = 'completed' THEN 1 ELSE 0 END) AS completed
```

## 十二、风险与缓解

| 风险                                     | 缓解措施                                                                                          |
| ---------------------------------------- | ------------------------------------------------------------------------------------------------- |
| SQLite schema 过于复杂，目标项目难以维护 | 所有表 DDL 拆为独立 SQL 文件，可选启用                                                            |
| 符号级 map 脚本在不同环境兼容性问题      | 零额外依赖原则 + 条件生成 + 优雅降级（符号级脚本失败时 fallback 到文件级 repo-map，不中断工作流） |
| 策略引擎过度拦截影响开发效率             | 默认策略偏宽松，仅 `.env`/secrets 为 high risk                                                    |
| Workflow preset 不适配所有项目           | preset 仅为参考，支持自定义 workflow 定义                                                         |
| MBTI 人格化导致 Agent 输出风格不一致     | 人格指令为建议性质，不覆盖核心功能指令                                                            |
| 设计范围大，实施周期长                   | 8 个 phase 独立可交付，可根据实际调整顺序                                                         |
| 红队测试误报率高，增加修复负担           | Round 1 聚焦高置信度系统化检查，Round 2 鼓励创意但需提供具体 payload 和复现步骤；评分标准明确     |
| 红队测试仅覆盖代码层面，不含真实渗透     | 明确定位为 AI 辅助代码级安全审查，非替代专业渗透测试；报告中注明"未覆盖区域"                      |
